Revised onboarding plan — v3

Based on the Critic's feedback from round 02 about decision fatigue in the 7-step flow, I've reorganized around three pressure zones: Warm-up, Cook, and Serve.

Zone 1 — Warm-up (1 screen)

  • Single question: "What are you hoping to cook up?" with four cards (Plan · Write · Decide · Explore)
  • No API key ask yet. Defer to first run.
  • Skippable via "I know what I'm doing"

Zone 2 — Cook (2 screens)

This is where the metaphor earns its keep. The user sees a live preview of the pot while they configure:

  • Model pair picker — Builder vs Critic, shown as two burners
  • Rounds slider labelled "how long to cook" (3–10)
  • Convergence threshold as the whistle pressure
Design principle: every config control should have a physical analogue on the pot. If a knob doesn't map, we cut it.

Zone 3 — Serve

First-run only: after round 1 completes, show a teardown overlay explaining what just happened. Subsequent runs skip this.

Removed from previous draft: the tutorial modal, the model-recommendation wizard, and the cost-preview gate. All three created friction without proportional value.

Where this still leaks pressure

The three-zone reframe is a real improvement — decision fatigue drops because each screen has a single intent. But I can push back on three points before we call this converged.

1. The "four cards" in Warm-up re-introduces the exact fatigue you claim to remove. Asking users to self-classify as Plan / Write / Decide / Explore is the same taxonomy tax as the old flow, just relabeled. Most users won't know which bucket they're in, and picking wrong sets a bad default. Consider: skip this screen entirely and let the topic textbox do the classifying via an embedding pass.

2. The burner metaphor is cute but inverted. In a real pressure cooker, both ingredients share one pot — not two burners. If Builder and Critic are the "ingredients," the UI should show them combining, not heating separately. A single pot with two labeled inputs would read cleaner.

3. The teardown overlay is a tutorial by another name. You removed the tutorial modal then added a post-round explainer that serves the same role. If the UI is legible, no teardown is needed. If it isn't, fix the UI.

What's actually landed

  • Deferring the API key ask — good. This was the biggest bounce point in v2.
  • The "how long to cook" slider is a clean metaphor collision. Keep it.
  • Removing the cost-preview gate: correct. Show cost live
